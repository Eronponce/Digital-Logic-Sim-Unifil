### Como usar o LoadingOverlay

```csharp
// Disparando
LoadingOverlay.Show("Sincronizando projeto...");

// Sempre esconde em TODOS os caminhos de saída (sucesso, erro, cancelamento):
LoadingOverlay.Hide();

// Draw() já é chamado no final de MainMenu.Draw(). Outros menus que existirem
// fora do MainMenu precisam chamar LoadingOverlay.Draw() também.
```

## 3. User stories — detalhamento para itens P1

### US-001: Mensagens de erro em português

> **Como** aluno brasileiro usando o simulador,
> **quero** ver mensagens de erro em português quando algo der errado no login,
> **para** entender o que fazer sem depender do professor traduzir Firebase errors.

Critérios de aceite:
- Erros comuns (email inválido, senha fraca, usuário não existe, senha errada, conta desabilitada, rede offline) têm tradução dedicada.
- Erros não mapeados caem em um fallback genérico ("Ocorreu um erro ao fazer login. Tente novamente.") sem expor o stack Firebase.
- Mensagem aparece na faixa de `statusMessage` do `LoginMenu` (já existente) em cor vermelha/laranja conforme severidade.

Implementação sugerida: helper `AuthErrorMessages.Localize(string firebaseError)` chamado dentro do `OnAuthError` em `LoginMenu`.

### US-002: Confirmação de Logout

> **Como** usuário logado,
> **quero** confirmação antes de fazer logout,
> **para** não perder acidentalmente a sessão com um clique errado.

Critérios de aceite:
- Botão "Logout" no menu principal abre popup modal.
- Popup tem botões "Cancelar" (default) e "Sair da conta".
- "Esc" cancela; "Enter" confirma apenas se "Sair da conta" estiver com foco.

### US-003: Enter / Tab no login

> **Como** usuário,
> **quero** poder submeter o formulário com Enter e navegar entre campos com Tab,
> **para** não precisar tirar a mão do teclado para clicar no botão.

Critérios de aceite:
- `Enter` em qualquer campo do Sign In dispara "Sign In".
- `Enter` no Create Account dispara "Create Account" apenas se o último campo estiver focado (evita submissão acidental).
- `Tab` navega na ordem: email → senha → (se Create Account) nome → matrícula → professor.

Bloqueador conhecido: precisa entender como o `InputField` do SebVis trata Tab/Enter hoje. Pode exigir ajuste no `UI.cs`.

### US-004: Lembrar último e-mail

> **Como** usuário recorrente,
> **quero** que o último e-mail usado apareça pré-preenchido,
> **para** não precisar digitar de novo toda vez que abrir o app.

Critérios de aceite:
- Após login com sucesso, salvar o e-mail em `PlayerPrefs` com chave `LoginMenu_LastEmail`.
- No `DrawFullLoginScreen`, se `email` está vazio e `PlayerPrefs` tem valor, preencher.
- Logout **não** apaga o e-mail salvo.
- Um checkbox opcional "Lembrar meu e-mail" (default ligado) permite desativar.

## 4. Notas técnicas

### Refactor do MainMenu
O `DrawMainScreen` hoje funciona com um enum `MainMenuAction` e dois arrays de ação (`loggedInActions` / `offlineActions`). Se um novo estado aparecer (ex: "modo convidado"), basta criar um novo array e adicionar casos no `HandleMainMenuAction`. Evita o bug silencioso de "troquei a ordem dos botões mas esqueci de trocar os índices".

### LoadingOverlay — limitações atuais
1. **Sem timeout**: se o callback de sucesso/erro nunca for chamado (bug no Firebase, rede morta sem erro), o overlay trava. Item #8 do P2 resolve.
2. **Não bloqueia input**: usuários podem apertar Esc/clicar fora e outros menus reagem. Precisa verificar se os menus abaixo respeitam `LoadingOverlay.IsVisible`. Hoje não respeitam.
3. **Singleton global**: dois consumidores simultâneos sobrescrevem a mensagem. Para o uso atual (sempre uma op Firebase por vez) está OK.

### Espaçamento do login
Os valores atuais de `ySpacing` (3.0 / 2.55 / 2.4) e `startYOffset` (13.0 / 16.8 / 17.4) foram ajustados **no chute**, sem rodar o build. Precisa validação visual nos 3 modos (Sign In, Create Account, Complete Profile) em pelo menos 2 resoluções (ex: 1280x720 e 1920x1080). Se ficarem apertados/largos, os números estão concentrados no método `GetLayout` de `LoginMenu.cs` e podem ser iterados isoladamente.
